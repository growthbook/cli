# Releasing the GrowthBook CLI

How a spec change becomes a published CLI, and what a reviewer needs to do. This
CLI is **generated by Speakeasy from the GrowthBook OpenAPI spec** (see
[CLAUDE.md](CLAUDE.md)); releases are driven off that spec, not hand-cut.

## The flow at a glance

```
growthbook: spec.yaml change on main
        │  (daily cron, or repository_dispatch)
        ▼
cli: Speakeasy regenerates → opens a version-bump PR   (gen.yaml versioningStrategy: automatic)
        │
        ▼
human review: confirm the major/minor/patch classification   (CODEOWNERS auto-requests you)
        │  approve + merge
        ▼
auto-tag: push v<version>   (.github/workflows/auto-tag.yaml, via RELEASE_TAG_PAT)
        │
        ▼
release: GoReleaser builds binaries + GitHub Release, then npm shim publishes   (release.yaml)
```

The **git tag `v<version>` is the single source of truth** for what gets published.
Everything downstream (binary version, npm version, dist-tag) is derived from it.

## Versioning

- **Independent semver, starting at 1.0.0.** The CLI version is *not* tied to the
  GrowthBook app version (the app is 4.x; the CLI is 1.x). They version separately.
- **The bump tracks the spec**, classified automatically by Speakeasy:
  - **major** — a breaking change (endpoint/field removed or renamed, a new required
    input, a type change).
  - **minor** — additive (new endpoint, new optional input, new output field).
  - **patch** — codegen/template/bugfix only, no spec change.
- **dist-tags** are derived from the tag in `release.yaml`: a prerelease tag
  (`v1.0.0-next.3`) publishes to **`next`**; a plain tag (`v1.2.0`) publishes to **`latest`**.

### Command versioning is part of this

Each command group points at the **newest** version of its API endpoint (e.g. `features` = v2
today); superseded versions stay available as `<group>-vN` (e.g. `features-v1` — deprecated, warns
on stderr). When a newer API version lands, the base group advances to it and the previous version
becomes the next `-vN`.

**Re-pointing a base command to a newer endpoint version is a breaking change → a MAJOR bump, and it
must be called out explicitly in the changelog.** Scripts invoking the bare `features` will start
receiving the new version's response shape, so customers need the heads-up to pin the prior major or
switch to the `-vN` command. Speakeasy's spec-diff won't necessarily classify this as major on its
own (the underlying endpoints both still exist) — **this is a case a reviewer must catch and force
with `set_version`** if the bump comes through as minor.

> Note: most spec changes don't strictly *require* a release to be usable — JSON/`--jq`
> output passes new fields through losslessly, and new request-body fields can go via
> `--body` raw JSON. The hard case that needs a release is a brand-new **endpoint** (no
> command exists yet). We still release on additive changes so the typed commands/flags
> and docs catch up.

## Upgrade prompts users see

A released version isn't just published — it's what the CLI uses to nudge users to
upgrade. The advisory (`internal/customcfg/updatecheck.go`) is **why the bump you confirm
matters**, so reviewers should know how it behaves:

- **Trigger:** the latest **stable** GitHub Release (prereleases are excluded) compared by
  semver against the running binary. So `@next` builds never nudge anyone, and a release
  goes live as a prompt the moment a plain `vX.Y.Z` tag exists.
- **Only on minor or major.** A **patch** bump is silent — patch is for codegen/no-capability
  changes, so users aren't pinged. **minor/major** bumps prompt. *This is the direct link to
  your review:* mis-labeling a real feature as a patch means users are never told to upgrade
  for it; mis-labeling a breaking change as minor under-signals it.
- **Once per version (backoff).** Each distinct newer version is mentioned at most once — an
  ignored prompt doesn't nag daily; the next prompt only comes with the next new version.
- **Only to servers that can use it (compat gate).** The prompt is suppressed if the
  connected server's build date is older than the new CLI's generated-against date — upgrading
  would point the CLI at endpoints the server lacks. If dates are unknown, it degrades to
  semver (permissive).
- **Quiet by construction:** stderr only (never pollutes stdout/JSON), at most once per 24h,
  off in CI and non-interactive shells, and opt-out via `--no-update-check` or
  `GBCLI_NO_UPDATE_CHECK`. The message points at `npm update -g growthbook`.

## For reviewers

When a generation PR is assigned to you, you are confirming **one thing above all: is
Speakeasy's version bump correct?**

1. **Read the PR title/body** — it states the proposed version and summarizes the spec
   changes (e.g. `1.2.0 → 1.3.0 (minor)`).
2. **Sanity-check the classification against the spec diff:**
   - Anything removed, renamed, made required, or retyped → this should be a **major**.
     If Speakeasy called it minor/patch, that's the case to catch.
   - Purely additive (new endpoint/field/optional input) → **minor** is right.
   - No spec change (codegen/dependency bump only) → **patch**.
   - You do **not** need to review the generated Go line-by-line — it's machine-output.
   - Remember the bump drives the user-facing upgrade prompt (see above): **patch is silent**,
     minor/major prompt compatible servers. Don't bury a real feature in a patch.
3. **If the bump is wrong, don't hand-edit the version.** Re-run generation with an
   explicit version so gen.yaml/gen.lock stay consistent:
   - GitHub UI: **Actions → Generate → Run workflow**, set **`set_version`** to the
     correct value (e.g. `2.0.0`).
   - or locally: `speakeasy bump major` (or `minor`/`patch`/`-v X.Y.Z`), then commit.

   The override replaces the bot's PR; review the new one.
4. **Approve + merge.** Merging is what triggers the tag and publish — so only merge when
   you're ready for that version to ship.

If a change *should* have been breaking but the spec was written in a backward-compatible
way (or vice-versa), fix it at the source (the Zod validators in `growthbook`) — the bump
only reflects what the spec says.

### Patches & name overrides — two extra things to check on generation PRs

Beyond the version bump, generation PRs can silently regress two hand-maintained layers:

- **`.speakeasy/patches/`** carries behavioral fixes to generated code (pagination hint,
  stdin `--body -`, output newlines, agent-mode error classing, …) that re-apply on every
  regen. A patch that no longer applies fails the generation — if the regen PR is missing or
  the workflow failed, check for patch conflicts first and rebase the affected `.patch`
  files. Prefer upstreaming fixes to Speakeasy over growing this directory (tracking issues
  #24–#29); each patch is recurring regen-conflict surface (the `--usage` KDL patch was
  already abandoned for this reason).
- **Verb renames** live in the overlay as per-op `x-speakeasy-name-override`s (e.g.
  `feature-revisions publish`). They are per-operation, so a **new endpoint added to a
  renamed group surfaces under its raw operationId** (`post-feature-revision-…`) until an
  override is added. When the spec diff shows a new operation in such a group, add the
  override in the same PR.

## Mechanics (files involved)

| Concern | Where |
|---|---|
| Regenerate + open PR (daily cron / dispatch) | `.github/workflows/sdk_generation.yaml` |
| Automatic bump classification | `.speakeasy/gen.yaml` → `versioningStrategy: automatic` |
| Auto-request a reviewer | `.github/CODEOWNERS` |
| Tag on merge → trigger release | `.github/workflows/auto-tag.yaml` |
| Build binaries + GitHub Release + npm publish | `.github/workflows/release.yaml`, `.goreleaser.yaml`, `npm/` |

### Required secrets & variables

| Name | Type | Purpose |
|---|---|---|
| _(none for npm)_ | — | npm publishes via **OIDC trusted publishing** — no token. `release.yaml` grants `id-token: write`; the npm↔GitHub trust relationship is configured on npmjs.com. |
| `RELEASE_TAG_PAT` | secret | push the tag from `auto-tag.yaml` so `release.yaml` fires (see gotcha) |
| `SPEAKEASY_API_KEY` | secret | generation |
| `RELEASE_AUTOMATION` | variable | set to `true` to arm `auto-tag.yaml` (off during `@next` staging) |

> **npm OIDC notes:** trusted publishing needs npm ≥ 11.5.1 (newer than Node 20's bundled npm), so
> the publish step runs `npm install -g npm@latest` first. The npm trust relationship is scoped to a
> specific workflow — **when we move to the `auto-tag` → `release` flow at GA, the trust config must
> be updated** to match (owner action).

## Manual / prerelease releases

Automatic versioning emits **stable** semver, so prereleases are cut by hand. This is the
path during `@next` staging (keep `RELEASE_AUTOMATION` unset so auto-tag stays dormant):

```bash
git tag v1.0.0-next.0
git push origin v1.0.0-next.0          # → release.yaml → publishes to the `next` dist-tag
./scripts/smoke-npm.sh 1.0.0-next.0    # after the release builds: pack/install/run check
npm view growthbook dist-tags          # confirm
```

A human-pushed tag triggers `release.yaml` directly (no PAT needed — the recursion block
below only applies to tags pushed *by a workflow*).

At GA cutover: push the current version as a **plain tag** (no `-next` suffix, e.g.
`v1.2.0` — the line has moved past 1.0.0 during staging), which publishes to `latest`;
then deprecate the old line:
`npm deprecate 'growthbook@<0.3' "v1 is a rewrite; see MIGRATION.md"`.

## Gotchas

- **A tag pushed by `GITHUB_TOKEN` does NOT trigger another workflow.** GitHub blocks
  recursive workflow runs, so `auto-tag.yaml` must push with `RELEASE_TAG_PAT` (a PAT with
  `contents: write`, or a GitHub App token) — otherwise `release.yaml` never fires.
- **`auto-tag.yaml` is armed by a variable** (`RELEASE_AUTOMATION == 'true'`). It is dormant
  until you flip it, so the automation can be committed without risking a premature release
  during `@next` staging.
- **The release must exist before the npm shim publishes** — the shim's postinstall
  downloads the `v<version>` GitHub Release asset. `release.yaml` enforces this with
  `npm-publish` → `needs: goreleaser`.
- **Bot-authored PRs don't trigger other workflows** either — if PR CI must run on the
  generation PR, the generation action also needs a PAT.
